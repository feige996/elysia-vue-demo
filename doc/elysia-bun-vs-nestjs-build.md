# Elysia+Bun 与 NestJS 构建差异说明

本文说明为什么当前项目可以将后端打包为单文件 `index.js`，而 NestJS 默认通常不是这种产物形态。

## 1. 结论先行

- `Elysia + Bun`：默认更容易产出单文件后端构建物
- `NestJS`：并非不能单文件，但默认构建链路通常输出多文件 `dist`

这主要是框架设计取向和默认构建策略不同。

## 2. 为什么 Elysia+Bun 更容易单文件

- 框架轻量，运行时“魔法”较少，静态分析负担低
- 业务组织偏显式依赖注入与路由组合，打包器更容易追踪依赖图
- `bun build` 对 TS/ESM 的后端打包路径直接，单入口体验好

在本项目中：

- 后端构建脚本：`api/package.json -> build`
- 产物：`api/dist/index.js`

## 3. 为什么 NestJS 默认多文件

NestJS 默认更强调稳定和通用性，常见构建链路偏向 `tsc` 产物：

- 大量使用装饰器与反射元数据（`reflect-metadata`）
- 常见模块扫描、动态模块、动态 Provider 等运行时机制
- 对静态打包与激进 tree-shaking 不如轻框架友好

因此默认通常是多文件 `dist`，以保证行为可预期。

## 4. NestJS 能不能打单文件

可以，但通常需要额外配置（如 webpack / esbuild / ncc 等），并处理：

- 反射元数据保留策略
- 动态 `require` 与资源文件
- 第三方包 external 策略
- 运行时路径与别名映射

所以是“可做，但不是默认最省心路径”。

## 5. 工程取舍建议

- 追求快速、轻量、单文件部署体验：优先 `Elysia + Bun`
- 追求大型团队规范、模块化能力、生态一致性：可选 `NestJS`

两者都能做生产，只是默认构建与维护成本侧重点不同。
