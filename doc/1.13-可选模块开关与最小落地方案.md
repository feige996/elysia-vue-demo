# 1.13 可选模块开关与最小落地方案

## 背景与目标

当前基座定位是“稳定、可复用、可演进”的工程底座，而非默认全功能后台产品。  
当队列、定时任务、邮件、多存储等增强能力在不需要时仍初始化运行，会带来额外资源与维护成本。

本方案目标：

1. 默认最小集可运行（核心鉴权/RBAC/基础业务链路）。
2. 增强能力通过配置开关按需启用。
3. 关闭模块时不注册路由、不初始化依赖、不占用资源。
4. 保持现有开发体验与 CI 质量门禁不退化。

## 为什么不是“默认全部开启”

- 资源成本上升：空跑 Redis/BullMQ/Cron/SMTP 等会增加内存、连接数、CPU 周期。
- 故障面扩大：可选依赖异常可能影响启动、健康检查和排障效率。
- 运维复杂度提高：监控、告警、密钥、配置项膨胀。
- 本地与 CI 负担增加：更多外部组件依赖导致启动和调试链路更重。

结论：**支持全部能力是必要的，但默认全部运行不是最优策略。**

## 设计原则

- 单一入口：模块装配只允许在 `api/src/app/module-registry.ts`。
- 单一配置：开关定义只允许在 `api/src/shared/config/env.ts`。
- 默认关闭增强模块：核心模块常开，增强模块默认 `false`。
- 可观测：`/ready` 明确返回模块启用状态与健康状态。

## 开关设计（第一期）

建议新增环境变量（默认值均为 `false`）：

- `FEATURE_QUEUE_ENABLED`
- `FEATURE_CRON_ENABLED`
- `FEATURE_MAIL_ENABLED`
- `FEATURE_STORAGE_EXTENDED`
- `FEATURE_MONITOR_ENABLED`

并在 `env.ts` 导出统一对象：

- `features.queue`
- `features.cron`
- `features.mail`
- `features.storageExtended`
- `features.monitor`

注意：

- 业务代码禁止直接读取 `process.env`。
- 仅允许读取 `env` / `features`。

## 模块装配机制

新增模块注册器 `api/src/app/module-registry.ts`，以声明式方式管理模块：

- 模块 id
- 是否启用（根据 `features`）
- 挂载方法（`app.use(...)`）

`api/src/app/index.ts` 改造为：

1. 先注册通用中间件和基础路由。
2. 再执行模块注册器进行条件挂载。
3. 记录启用模块清单（用于日志与诊断）。

## 健康检查约定

`/ready` 保持统一返回结构：

- 模块关闭时：`enabled: false`, `ok: true`, `detail: "disabled by feature flag"`
- 模块开启时：执行真实健康检查并返回结果

这样可区分“不可用”与“未启用”，避免误报警。

## 本地验证（以 monitor 模块为例）

默认情况下 `FEATURE_MONITOR_ENABLED=false`，`/api/monitor/features` 不会被挂载（返回 `404`）。

### 1) 默认关闭时验证

```bash
# 启动 API（默认 monitor 关闭）
bun run --cwd api dev

# 另一个终端请求
curl -i http://localhost:3000/api/monitor/features
```

预期：`HTTP/1.1 404 Not Found`

### 2) 打开开关后验证

```bash
# 启动前临时注入环境变量（macOS / Linux）
FEATURE_MONITOR_ENABLED=true bun run --cwd api dev

# 另一个终端先登录拿 token
TOKEN=$(
  curl -s -X POST http://localhost:3000/api/auth/login \
    -H "Content-Type: application/json" \
    -d '{"account":"admin","password":"admin123"}' \
  | jq -r '.data.accessToken'
)

# 带 token 访问 monitor 路由
curl -s http://localhost:3000/api/monitor/features \
  -H "Authorization: Bearer $TOKEN" \
  | jq
```

预期：

- HTTP 状态 `200`
- 返回 `data.features.monitor = true`
- `/ready` 的 `checks.monitor.enabled = true`

### 3) 自动化验证

```bash
bun run --cwd api test
```

已覆盖用例：开关关闭返回 `404`，开启返回 `200`。

## 最小可落地 PR 方案（MVP）

### 目标

在不引入新业务模块的前提下，先完成“可选模块开关框架”落地，确保后续接入增强模块时无需二次改造主干。

### 计划改动文件

1. `api/src/shared/config/env.ts`
   - 新增 feature flag 环境变量解析与 `features` 导出。
2. `api/src/app/module-registry.ts`（新增）
   - 定义模块元数据与条件挂载逻辑。
3. `api/src/app/index.ts`
   - 替换硬编码 `.use(module)` 串联方式，改为注册器装配。
   - 打印已启用模块日志。
4. `api/tests/unit/module-registry.test.ts`（新增）
   - 校验开关开/关下的装配行为。
5. `doc/1.10-当前状态与验收口径.md`
   - 增加“可选模块开关框架”状态跟踪项。
6. `doc/1.12-路线图与Backlog.md`
   - 增加“增强模块按开关接入（队列/邮件/监控）”任务项。

### 不在本 PR 范围内

- 不引入 BullMQ/Cron/SMTP 等新增依赖。
- 不新增业务模块路由。
- 不调整现有 API 契约。

## 验收标准

功能验收：

1. 默认配置下，现有功能行为与接口返回不变。
2. 模块注册路径由硬编码切换为配置驱动，且可输出启用模块列表。
3. `/ready` 可正确区分模块未启用与异常不可用。

工程验收：

1. `bun run --cwd api typecheck` 通过。
2. `bun run --cwd api test` 通过。
3. 根目录 `bun run dev:check` 通过。

## 风险与回滚

潜在风险：

- 模块装配顺序变化导致中间件链路行为差异。
- 新增配置解析影响环境变量校验。

回滚策略：

- 保留旧挂载逻辑在独立 commit（或快速回滚到上一版本）。
- 若出现线上问题，优先回退到硬编码模块挂载版本。

## 第二期建议（接入增强模块）

优先级建议：

1. 监控模块（低风险、可见性高）
2. 邮件能力（边界清晰）
3. 队列与定时任务（依赖较多，放后）

每接入一个增强模块，必须同时补齐：

- 开关配置
- 健康检查
- 最小单元测试
- 文档更新
