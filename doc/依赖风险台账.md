# Harbor 依赖风险台账（最小版）

> 目标：把依赖安全问题从“发现”变成“可跟踪、可决策、可验收”。
>
> 维护节奏：每周至少更新 1 次；每次发布前复核 1 次。

## 1. 使用说明

- 风险来源：`bun audit`、依赖发布公告、CI 安全扫描。
- 状态定义：
  - `待处理`：已确认风险，尚未执行措施。
  - `处理中`：已启动修复或替代，尚未验证完成。
  - `已完成`：修复并通过回归验证。
  - `暂缓`：短期不修复，但有明确风险说明、缓解措施和复评时间。
- 建议优先级：
  - `P0`：high/critical，且可触达核心链路（登录、鉴权、数据写入、文件上传）。
  - `P1`：medium，或 high 但有较强外围缓解措施。
  - `P2`：low，或仅开发环境影响。

## 3. 执行清单（每个风险项）

1. **确认事实**：记录漏洞编号/公告链接、可利用条件、影响范围。
2. **制定方案**：升级、替代、移除、暂缓（含缓解措施）。
3. **执行变更**：升级依赖并修复兼容问题。
4. **回归验证**：
   - `bun run dev:check`
   - `bun run test`
   - `bun run build`
   - 必要时执行 `bun run test:e2e:admin`
5. **关闭项更新**：填入 PR/commit、验证结果、关闭时间。
