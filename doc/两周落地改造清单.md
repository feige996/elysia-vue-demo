# 两周落地改造清单

目标：先稳住可运行与可维护，再逐步提升文档与工程质量。  
周期：10 个工作日（2 周）。  
优先级：`P0` 必须完成，`P1` 建议完成，`P2` 有余力再做。

## P0（第 1 周，必须完成）

### 1) 配置与环境收口（Day 1）
- 目标：消除环境不一致与端口混乱风险
- 改造点：
  - 统一端口与 API 基地址读取规则（`api=9000`）
  - 校验 `.env/.env.example/docker-compose/CI` 一致性
- 文件：
  - `api/src/shared/config/env.ts`
  - `api/.env.example`
  - `admin/.env.example`
  - `web/.env.example`
  - `docker-compose.yml`
  - `.github/workflows/ci.yml`
  - `README.md`

### 2) 登录链路稳定性加固（Day 1-2）
- 目标：避免登录后路由递归、状态竞争
- 改造点：
  - 守卫中“初始化状态 + 动态路由注册”幂等化
  - `bootstrapAuthContext` 增加 in-flight 锁（Promise 复用）
- 文件：
  - `admin/src/router/index.ts`
  - `admin/src/store/auth.ts`
  - `admin/src/views/LoginPage.vue`

### 3) 请求层防循环刷新（Day 2-3）
- 目标：避免 refresh 触发链式递归导致栈溢出
- 改造点：
  - `authRequest` 仅在明确 401 时尝试刷新
  - 刷新接口调用强制走 `request`（不走 `authRequest`）
  - 最大重试次数限制为 1
- 文件：
  - `shared/request/eden.ts`
  - `shared/request/eden.test.ts`
  - `admin/src/api/request.ts`
  - `web/src/api/request.ts`

### 4) 菜单树构建防环保护（Day 3）
- 目标：后端遇到脏数据不爆栈
- 改造点：
  - `toMenuTree` 增加 visited 检测
  - 发现环时降级处理并记录日志
- 文件：
  - `api/src/modules/user/user.repository.ts`
  - `api/src/shared/logger/log.service.ts`

### 5) 回归测试补齐（Day 4-5）
- 目标：把近期问题沉淀为自动化防线
- 改造点：
  - admin 最小 e2e：登录 -> 拉权限/菜单 -> 进入首页
  - request client 单测：401 刷新成功/失败
  - API 集成测试覆盖 `/permissions/current`、`/menus/tree`
- 文件：
  - `shared/request/eden.test.ts`
  - `api/tests/integration/http.integration.test.ts`
  - `admin/tests/*`（可新增）

## P1（第 2 周，建议完成）

- 目标：避免文档与代码再漂移
- 改造点：
  - CI 保持 `openapi:generate + git diff` 漂移检查

- 文件：

  - `api/package.json`

### 8) 错误码与错误结构统一（Day 8）
- 目标：前后端错误处理可预测
- 改造点：
  - 统一 `code/message/requestId` 语义

- 文件：

  - `api/src/app/middleware/index.ts`

  - `admin/src/views/Login.vue`

### 9) 数据与种子脚本健壮性（Day 9-10）
- 目标：新成员一键启动不踩坑

- 文件：
