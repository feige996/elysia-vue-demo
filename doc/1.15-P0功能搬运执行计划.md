# 1.15 P0 功能搬运执行计划

## 目标

基于 `1.14-对标Elysia-Admin差异与功能搬运清单`，将 Harbor 的 P0 搬运项落为可执行任务，确保：

- 任务可拆分到具体文件与接口。
- 每项有最小验收与测试口径。
- 继续遵循 Harbor 的“默认最小集 + feature flag 按需启用”原则。

本期范围仅包含 P0：

1. 监控任务中心（任务列表 + 启停 + Cron 管理）
2. 部门管理与数据权限基础能力
3. 字典管理后台化（补齐写操作）

## 当前完成进度（2026-04）

- 已完成：`P0-2` 部门管理与用户部门筛选（后端接口 + 管理端页面）。
- 已完成：`P0-3` 字典管理后台化（后端接口 + 管理端页面 + 按钮级权限显示）。
- 部分完成：`P0-1` 监控任务中心（已具备最小 CRUD/启停/手动触发链路，待补测试与队列化增强）。

## 迭代节奏（建议 2 周）

- 第 1 周：后端能力优先（接口、数据模型、权限点、审计日志）
- 第 2 周：管理端页面 + 联调 + 自动化测试 + 文档收口

## 共性约束（所有 P0 模块必须满足）

- 所有新增模块均通过 `api/src/shared/config/env.ts` 增加开关，并接入 `api/src/app/module-registry.ts`。
- 默认关闭：不开启时路由不挂载、`/ready` 返回 disabled 状态。
- 新增管理接口必须挂 RBAC 权限点并记录审计日志。
- 至少补齐 1 类自动化测试（单元或集成），建议关键路径做集成测试。

## A. 监控任务中心（P0-1）

状态：部分完成（最小闭环已可用，未达 DoD 完成态）

## A1 后端任务

- 已落地（最小可用）
  - 数据表：`sys_jobs`（含状态、执行计数、最近执行信息、软删除）。
  - 接口：`/api/monitor/jobs`、`/api/monitor/jobs/:id/toggle`、`/api/monitor/jobs/:id/run`。
  - 行为：支持任务新增/编辑/启停/手动触发与最近执行记录回写。
- 与原计划差异
  - 原计划路径写为 `/api/jobs`，现实现为 `/api/monitor/jobs`（已纳入 monitor 域）。
  - 当前“立即执行”为最小记录版，尚未接入真实队列 worker 执行。
- 待补齐
  - 接入 `features.queue/features.cron` 的真实调度与执行链路。
  - 统一“下一次执行时间（nextRunAt）”计算与返回。

## A2 前端任务（admin）

- 已落地：`admin/src/views/JobManagement.vue`
- 已支持：列表、新增/编辑、启停、立即执行、最近执行信息展示。
- 已落地权限点（命名采用 monitor 域）
  - `monitor:job:view`
  - `monitor:job:create`
  - `monitor:job:update`
  - `monitor:job:toggle`
  - `monitor:job:run`

## A3 测试与验收

- 当前结论
  - 手工联调已通过：新增 -> 编辑 -> 启停 -> 手动触发 -> 列表回显。
  - 自动化测试尚未补齐（API 集成 + 管理端 E2E 均缺）。
- 未完成验收项
  - “cron 配置变更后真实调度生效”尚未闭环（需队列/调度接入）。
  - 审计日志对任务操作的专项回归用例未补齐。

## B. 部门管理与数据权限基础能力（P0-2）

- 权限点

## B3 测试与验收

## C. 字典管理后台化（P0-3）

状态：已完成（后端接口 + 管理端页面 + 联动验证）

## C1 后端落地（已完成）

- 在现有 `api/src/modules/system/system.module.ts` 内落地字典写接口（未拆新模块，避免分散）
- 已提供 API
  - 字典类型：`GET/POST/PUT/DELETE /api/dict-types`
  - 字典项：`GET/POST/PUT/DELETE /api/dict-items`
  - 状态切换：`POST /api/dict-types/:id/toggle`、`POST /api/dict-items/:id/toggle`
- 已落地约束
  - 字典类型编码唯一（创建/编辑均校验重复）
  - 删除字典类型前检查字典项，存在时阻止删除
  - 字典项排序字段通过 schema 约束（`0-9999`）校验合法性

## C2 前端落地（admin，已完成）

- 已采用“主从一体页”方案：`admin/src/views/DictConfigManagement.vue`
  - 上半区：字典类型列表、启停、增删改
  - 中间区：按类型管理字典项（标签、值、排序、状态、默认）
  - 下半区：兼容旧读取接口预览（按字典编码读取），用于联动回归验证
- API 封装已落地：`admin/src/api/modules/dict-config.ts`
- 当前菜单与页面访问权限沿用 `system:dict:view`（页面级）

## C3 测试与验收（当前结论）

- 接口链路已具备
  - 新增类型 -> 新增项 -> 按类型查询项
  - 类型编码唯一约束与删除保护规则均已在接口层实现
- 前端联调能力已具备
  - 字典项修改后可在“字典读取预览”区域实时看到数据变化
- 验收结论
  - 字典维护已由后台 CRUD 接管，不再依赖手工 SQL
  - 具备可回放的管理操作链路（后续可继续增强到更细粒度权限与审计项）

## F. 风险与应对

- 队列/调度依赖引入后本地环境复杂度上升
  - 应对：默认关闭，提供最小本地启动说明
- 部门数据与现有用户/角色关联改造可能影响旧数据
  - 应对：迁移脚本提供默认部门兜底
- 字典后台化后写操作权限边界不清
  - 应对：先收敛到管理员角色，后续再做细粒度授权

## G. 完成定义（DoD）

- P0 三项能力均具备“接口 + 页面 + 权限 + 审计 + 测试 + 文档”六件套。
- 模块可通过 feature flag 独立启停。
- 核心演示路径可在 15 分钟内完整走通（登录 -> 配置/管理 -> 查询验证）。
