# 1.11 工程治理与发布规范

## 1. 合并前最低检查

在仓库根目录执行：

```bash
bun run dev:check
bun run test
bun run build
```

涉及后台关键链路变更时，追加：

```bash
bun run test:e2e:admin
```

## 2. 分支与提交约定

- 一次提交聚焦一个目标（功能、修复、文档）。
- 提交信息清晰表达“为什么改”，而不是仅描述“改了什么”。
- 涉及 API 契约变更时同步更新接口文档与调用侧。

## 3. 发布流程（最小版）

1. 合并前完成本地与 CI 回归。
2. 目标环境执行数据库迁移：`bun run --cwd api db:migrate`。
3. 发布后检查 `/ready`，确认依赖状态为就绪。
4. 抽检登录、权限、关键写操作链路。

## 4. 回滚原则

- 应用优先回滚到上个稳定版本。
- 数据结构变更仅通过迁移管理，不使用不可追溯方式改库。
- 回滚后再次验证 `/ready` 与关键业务链路。

## 5. 文档维护规范

- 对外介绍类文档：维护在 1.00~1.02。
- 内部治理类文档：维护在 1.10~1.12。
- 过程性、阶段性材料统一移动到 `doc/archive/`。

## 6. 风险治理联动

- 依赖风险统一在 `依赖风险台账` 维护，不在多个文档重复记录。
- 暂缓项必须包含：原因、缓解措施、复评日期、负责人。
