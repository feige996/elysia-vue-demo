# 1.14 对标 Elysia-Admin 差异与功能搬运清单

## 目标与范围

本文对比当前 Harbor 项目与 `elysia-admin` 项目在**产品能力、工程架构、可运维性、开发效率**四个维度的差异，输出：

1. 双方优缺点（按落地价值视角）。
2. Harbor 建议搬运的能力清单（含优先级）。
3. 最小落地顺序（避免一次性“大改造”）。

对比依据来自当前仓库代码与文档：

- Harbor：`README.md`、`doc/1.01-已完成功能清单.md`、`api/src/app/module-registry.ts`、`api/src/modules/system/system.module.ts`、`api/src/modules/monitor/monitor.module.ts`
- Elysia-Admin：`elysia-admin/README.md`、`elysia-admin/server/src/modules/`、`elysia-admin/server/src/core/task-registry.ts`、`elysia-admin/server/src/infrastructure/queue/core/scheduler.ts`、`elysia-admin/admin/src/views/`

## 一、总体差异（先看结论）

- Harbor 更像“可治理的业务基座”：强调最小可用、质量门禁、可演进开关化能力。
- Elysia-Admin 更像“功能完备的后台产品模板”：模块覆盖更广、页面更齐全、可直接演示的系统功能更多。
- Harbor 在工程规范与渐进演进策略上更稳；Elysia-Admin 在开箱即用业务功能上更强。
- 结论不是二选一：建议 Harbor 保持当前架构治理优势，同时按优先级吸收 Elysia-Admin 的高价值模块。

## 二、优缺点对比

## Harbor 的优势

- **工程治理成熟**：根目录已形成 typecheck/lint/format/test/build 的统一质量门禁，且有文档化协作体系。
- **最小集与开关化方向明确**：`module-registry` + `features` 已落地，支持增强模块按需挂载，降低资源占用与故障面。
- **基座能力聚焦核心链路**：认证、RBAC、审计、错误码、就绪检查已经打通，适合作为组织内多业务复用底座。
- **前后端一致性较好**：Monorepo 统一运行脚本，开发启动与 CI 对齐成本较低。

## Harbor 的不足

- **业务模块覆盖仍偏少**：相较 Elysia-Admin，系统管理与监控域尚有明显缺口（如部门、字典全量管理、任务调度中心等）。
- **可运维能力仍以基础监控为主**：当前已有在线/缓存/IP 黑名单等基础监控，但“任务队列可视化 + 调度治理”尚未形成完整闭环。
- **管理端页面丰富度不足**：当前管理端页面数量与交互丰富度明显少于 Elysia-Admin，演示说服力与“可直接商用感”偏弱。

## Elysia-Admin 的优势

- **模块覆盖广、即用性高**：系统管理（用户/角色/菜单/部门/字典/API/存储/IP 黑名单）+ 监控（在线/任务/缓存/日志）较完整。
- **任务调度链路完整**：从数据库恢复任务到队列并执行，`task-registry` + `scheduler` 已形成实际可运行链路。
- **前端管理体验更“产品化”**：页面与组件体系较完整，主题、布局、全局组件等配置化能力较突出。
- **部署选项更丰富**：文档层面给出普通构建、二进制、Docker、PM2 等多模式说明。

## Elysia-Admin 的不足

- **工程一致性复杂度更高**：前端基于 pnpm/node、后端基于 bun，跨环境心智与维护成本高于 Harbor 的统一工作区方案。
- **功能面更大带来维护压力**：模块多、依赖多，长期升级与安全治理成本会更高。
- **“默认全开”风险更高**：当增强能力未按需开关化治理时，资源占用和故障面会扩大（Harbor 现阶段在这点更克制）。

## 三、Harbor 建议搬运功能清单（核心输出）

## P0（优先搬运：短期高价值）

- **监控任务中心（任务列表 + 启停 + Cron 管理）**
  - 价值：补齐可运维闭环，直接提升生产可控性。
  - 参考：`elysia-admin/server/src/modules/monitor-job/`、`elysia-admin/admin/src/views/monitor/job/`
- **部门管理与数据权限基础能力**
  - 价值：补齐 RBAC 里的组织维度，支撑中后台常见“按部门隔离”诉求。
  - 参考：`elysia-admin/server/src/modules/system-dept/`、`elysia-admin/admin/src/views/system/dept/`
- **字典管理后台化（不仅是字典读取）**
  - 价值：减少硬编码，提升业务可配置化能力。
  - 参考：`elysia-admin/server/src/modules/system-dict/`、`elysia-admin/admin/src/views/system/dict/`

## P1（次优先：增强平台化能力）

- **API 管理（接口元数据/权限点可视化）**
  - 价值：让权限治理、接口审计更透明，便于新成员理解系统边界。
  - 参考：`elysia-admin/server/src/modules/system-api/`、`elysia-admin/admin/src/views/system/api/`
- **存储管理中心（多存储策略统一管理页）**
  - 价值：当前 Harbor 已有 local/oss/cos 能力，可补齐“可视化配置与审计”。
  - 参考：`elysia-admin/server/src/modules/system-storage/`、`elysia-admin/admin/src/views/system/storage/`
- **登录日志专题页（与操作日志并行）**
  - 价值：安全排障粒度更细，便于定位账号异常行为。
  - 参考：`elysia-admin/server/src/modules/system-login-log/`、`elysia-admin/admin/src/views/system/log/loginlog/`

## P2（选择性搬运：按产品定位取舍）

- **主题/布局/全局组件配置体系**
  - 价值：提升可定制体验，但对“基座能力”不是刚需。
  - 参考：`elysia-admin/admin/src/config/modules/component.ts`
- **业务示例模块（订单/商户/支付/退款）**
  - 价值：提升演示完整度，适合作为脚手架示例而非默认核心模块。
  - 参考：`elysia-admin/server/src/modules/business-*/`、`elysia-admin/admin/src/views/business/`
- **注册/忘记密码等开放认证流程**
  - 价值：面向 ToC 或开放平台时价值高；纯内部后台可后置。

## 四、建议“不直接搬运”的部分

- 不建议原样引入 Elysia-Admin 全量前端依赖栈（Element Plus + Tailwind + 大量生态库），避免破坏 Harbor 当前管理端技术选型与构建稳定性。
- 不建议一次性迁移其全部业务示例模块，优先抽象为“可插拔示例包”。
- 不建议在未完成 feature flag 对齐前引入队列/任务/邮件等强依赖模块，避免回到“默认全开”的老路。

## 五、最小落地路线（建议 3 个迭代）

- **迭代 1（P0）**：任务中心 + 字典管理后台化
  - 后端：新增 `monitor-job`、扩展 `system-dict` 的写接口
  - 前端：新增任务页、字典管理页
  - 治理：全部能力挂到 `features` 开关下
- **迭代 2（P0/P1）**：部门管理 + 登录日志页
  - 后端：`system-dept`、`system-login-log`
  - 前端：部门树、登录日志筛选/导出
  - 治理：补齐测试与 ready 状态暴露
- **迭代 3（P1）**：API 管理 + 存储管理中心
  - 后端：接口元数据聚合、存储策略管理 API
  - 前端：API 权限点管理页、存储配置页
  - 治理：完善权限点、审计日志与文档说明

## 六、验收标准（用于判断搬运成功）

- 新增模块必须支持 feature flag（默认关闭，按需启用）。
- 新增模块必须补齐最小测试（至少单元或集成一类）。
- 新增模块必须接入审计日志与权限点。
- 文档必须同步更新（README + `doc/README.md` + 模块说明）。

## 七、一句话结论

Harbor 应继续保持“基座优先、渐进增强”的路线：以 Elysia-Admin 为功能能力池，优先搬运任务中心、部门管理、字典后台化这三类高回报能力，而不是追求一次性功能对齐。
